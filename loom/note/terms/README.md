# Terms

The definition surface of the method. Every term that does load-bearing work across more than one file is its own artifact here, grouped by sub-directory, in a vocabulary that does not assume Loom or any particular adopter.

**Scope of validity.** These definitions apply within Note-tagged and Sett-tagged contexts: this corpus, the requirements and specifications of the method and the tool, and any file authored as part of the corpus. A project adopting Note may use the same words differently in its own domain content; within project-tagged contexts the project's meaning prevails. Two conventions support disambiguation: a term's meaning is determined by the artifact it appears in; and where a collision would cost real work, link directly to the term entry, which is authoritative. Term recognition in prose follows the Title Case rule fixed by [The spine pack](../specifications/note/spine-pack.md){id=note:spec:6kjja5r}.

**What this is not.** Not the methodology (how work is done lives in the specifications). Not the authoring rules (the prose conventions live in `../../meta/prose-conventions.md`, and the rules an artifact must satisfy live in the requirements and specifications). Not a dictionary of common English; only terms-of-art whose meaning is not obvious from common usage earn an entry.

**Grounding.** A term earns an entry when it is load-bearing across multiple files; single-use jargon stays in its file. Origin sections cite the relevant Oxford sense to ground the term; where a term is a purely technical coinage, the academic lineage is optional, but if no Oxford sense plausibly maps, the term is probably stretched and should be reconsidered.

Each term is its own artifact under a group sub-directory, and the directory is the grouping. The per-group relation diagrams are derived views that Sett generates, not authored in the corpus.
