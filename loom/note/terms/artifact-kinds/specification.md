A precise, committed description of a designed solution element, in enough detail that one actor can implement it and another can check the implementation against it.

Unlike a Requirement, a Specification *commits to a design*: it makes choices, adopts formats, and names mechanisms.

A Specification satisfies one or more Requirements.

## Inclusion test

Is it a precise description of how an element is designed, detailed enough for independent implementation and verification?

If it states what must be true without committing to a design, it is a Requirement.

## Relations

- *satisfies* [Requirement](requirement.md){id=note:term:tr9zna6}

## Origin

General English (Oxford: "a detailed description of the design and materials used to make something").

Used as an artifact kind through IEEE 830 and its successor ISO/IEC/IEEE 29148:2018, and in the MBSE tradition where the satisfy relationship binds a requirement to the specification that fulfills it.
